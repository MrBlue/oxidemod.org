    This isn't exactly true. The OS starts swapping before (when) running out of memory.
    If you're using Linux and don't have a swap partition then you might be in trouble, when the swap partition is full as well, something blows up (in two ways, depending on your configuration: either a process is killed by the kernel or malloc calls will fail in all processes, causing errors as well).
    Assuming that only that single server is using 8-10GB and the other servers also require some RAM, then this might be happening.
    I also doubt you've got 20GB of RAM (unless you're hosting on a VPS?), that would be a very uncommon combination.
    Either way, increase the size of your swap partition to "fix" this issue. Then again, the only way to REALLY fix this issue is to reduce the load on your hardware, since swapping is pretty expensive and once you start swapping each memory access will be loads slower.

    A nice comparison for average values:
    1 CPU cycle 0.3 ns
    Level 1 cache access 0.9 ns
    Level 2 cache access 2.8 ns
    Level 3 cache access 12.9 ns
    Main memory access 120 ns
    Solid-state disk I/O 50-150 μs
    Rotational disk I/O 1-10 ms

    Normal RAM access is between level 1 and main memory (although since this is unity, you'll probably only get main memory hits), while swap partition access might be between SSD and HDD.
